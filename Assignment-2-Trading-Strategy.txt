B.2 How the Backtest Works
The backtesting framework evaluates the strategy once per week, every Friday. No price monitoring or trade execution occurs on any other day. The following sequence occurs at each evaluation date:

1. Evaluate all stocks: The strategy examines the latest price and indicator values for all stocks in the dataset.
2. Generate signals: For each stock, the strategy logic returns a BUY, SELL, or HOLD signal.
3. Execute trades: Stocks signalled BUY are purchased (if not already held). Stocks signalled SELL are liquidated. All remaining positions are held unchanged.
4. Record portfolio value: The total value of all holdings plus cash is recorded for that Friday.
No Intra-Week Trading
The simulation executes only on Fridays. There is no monitoring or trading between weekly evaluation dates. If a stock crosses a stop-loss threshold on a Wednesday, the strategy will not react until the following Friday. This is a deliberate design constraint that ensures consistency across all evaluations.
Practical details:
Cash: Uninvested capital remains as cash and is available for new positions the following week.
Position size: By default, each new position targets a fixed amount per stock. This may be modified in the enhanced strategy.
Transaction costs: The baseline assumes zero transaction costs. These may optionally be incorporated to improve simulation realism.

C.1 Exploratory Data Analysis
The provided notebook includes a basic EDA as a starting point. Consider what additional analysis would give a clearer understanding of the data before any strategy decisions are made. What patterns exist in price behaviour, volume, or transcript characteristics? What anomalies or data quality issues might affect results? The findings from this phase should directly inform the choices made in subsequent areas.
C.2 Data Cleaning and Preparation
Raw financial data often contains gaps, anomalies, and inconsistencies that can distort indicator calculations and backtest results. Consider how missing price values should be handled, whether all stocks have sufficient historical coverage to be included, and whether any observations appear to be erroneous. Also consider whether the composition of the dataset introduces any systematic bias in historical performance estimates.
C.3 Technical Indicators and Features
The baseline relies solely on a 50-day moving average. A range of additional indicators exists that capture different aspects of price behaviour - momentum, mean reversion, volatility, and volume dynamics, among others. Consider which of these might be informative for the trading signals, how their parameters should be set, and whether combining multiple indicators leads to more reliable signals than any single one in isolation.
Note: Computing indicators across a large number of stocks and many years of data can be computationally expensive when implemented iteratively. Consider how vectorised or parallelised approaches might improve performance, and document the impact.
C.4 Better Use of Earnings Transcripts
The baseline extracts a fixed excerpt from each transcript and produces a single sentiment label. Consider finding the most informative sections of the transcript, confidence score returned by FinBERT to weight the signal, and role of neutral sentiment in decision-making. Think about how sentiment patterns across multiple quarters might add further predictive value.
C.5 Entry, Exit, and Risk Rules
The baseline uses a single indicator for entry and a fixed stop-loss for exit. Consider whether requiring agreement across multiple signals before entering a trade improves outcome quality, how position sizing might be adjusted to reflect signal confidence, and whether alternative exit conditions such as a tighter stop-loss or a profit target, would improve overall performance. Also consider how concentration risk across positions should be managed.

D. Scalability and Computational Efficiency
With approximately two million price records in the development set, naive iterative implementations are likely to be slow and may encounter memory constraints. To handle this you could experiment with vectorization and parallelization. Your solution should be able to handle larger dataset than current size without significant performance degradation. Scalability is highly valued therefore design your solution to be robust.
E. Performance Metrics Reference
The following metrics are used to evaluate strategy performance:

| Metric | Description | Target |
| --- | --- | --- |
| Total Return | Overall percentage change in portfolio value from start to end of the evaluation period. | Maximise |
| Sharpe Ratio | Return per unit of risk (volatility). Higher values indicate better risk-adjusted performance. A value above 1.0 is generally considered favourable. | >1.0 |
| Maximum Drawdown | The largest peak-to-trough decline in portfolio value before recovery. Indicates the worst-case loss experienced during the period. | Minimise |
| Win Rate | The proportion of individual trades closed at a profit. | >40% |
| Volatility | Annualised standard deviation of weekly portfolio returns. Measures the stability of returns over time. | Lower for equivalent return |

The development set spans a significantly longer evaluation horizon than the validation set. The higher cumulative return observed on the development set is therefore expected and should not be interpreted as evidence of superior strategy performance. For cross-period comparison, the Sharpe ratio can be referred, it is annualised and accounts for the difference in evaluation window.
